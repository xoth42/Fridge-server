Deployment notes — hosting patterns, Docker, and production checklist

Minimum environment variables
- `SLACK_BOT_TOKEN` (xoxb-...)
- `SLACK_SIGNING_SECRET`
- `SLACK_APP_TOKEN` (xapp-... for Socket Mode)
- `PORT` (for web mode)

Two common deployment modes
- Web mode (HTTP endpoints): host a TLS endpoint (behind Caddy/nginx). Set Interactivity & Event Subscription Request URLs to your public HTTPS endpoints.
- Socket Mode: use Bolt's `SocketModeHandler` with `SLACK_APP_TOKEN`. Avoids exposing a public HTTPS endpoint — preferred when hosting inside private network or dev machines.

Example Docker Compose snippet (env-driven)
  services:
    slackapp:
      image: myorg/slackapp:latest
      environment:
        - SLACK_BOT_TOKEN=${SLACK_BOT_TOKEN}
        - SLACK_SIGNING_SECRET=${SLACK_SIGNING_SECRET}
        - SLACK_APP_TOKEN=${SLACK_APP_TOKEN}
        - PORT=3000
      restart: unless-stopped

Secrets & operational practices
- Use a secrets manager (HashiCorp Vault, cloud KMS/Secret Manager) for token storage; do not commit `.env` in repo.
- Handle SIGTERM gracefully so in-flight acknowledgements / web requests complete or are requeued.
- Use health checks and liveness probes; autoscale consumers for high event volumes.

Logging & observability
- Log Slack `event_id`, `action_id`/`trigger_id`, user/workspace IDs, API errors, and `Retry-After` headers.
- Export metrics for: request rate, ack latency, 429 counts, failed verifications.

Security & networking
- Use TLS (Let’s Encrypt or managed TLS). Slack requires valid certs for webhook delivery.
- If using web mode behind a proxy, forward raw request body for signature verification (avoid body re-writes).
